## 一、MVP（最小可行产品）定义

### 1.1 MVP 核心目标

**让用户能够在 5 分钟内完成一次自助故障诊断，并获得可操作的解决方案。**

### 1.2 MVP 交付范围

| 模块 | 功能 | 优先级 |
|------|------|--------|
| **急诊入口** | 首页症状选择器 | P0 |
| **智能诊断** | 基于规则的故障排查引擎 | P0 |
| **方案库** | 常见问题解决方案展示 | P0 |
| **问诊预约** | 提交工单 + 联系方式收集 | P1 |
| **知识库** | 文档阅读（精简版） | P2 |

### 1.3 MVP 不包含

- ❌ 实时聊天客服
- ❌ 会员系统
- ❌ 支付功能
- ❌ 企业后台
- ❌ 上门服务预约
- ❌ 视频诊断

### 1.4 MVP 验证假设

1. **假设 1**：OpenClaw 用户存在可诊断的常见故障模式
2. **假设 2**：用户愿意通过自助工具解决简单问题
3. **假设 3**：用户愿意为复杂问题预约人工服务
4. **假设 4**：口碑传播是主要获客渠道

### 2.1 功能优先级矩阵

```
                    高价值

                      │
    ┌─────────────────┼─────────────────┐
    │   快速迭代      │   核心壁垒      │

    │                 │                 │
```

### 2.2 详细功能清单

#### P0 - 必须有（MVP）

| 功能 | 描述 | 预期工时 |
|------|------|----------|
| **症状选择器** | 用户选择 Agent 出现的症状（启动失败、回复异常、响应慢等） | 2天 |
| **诊断引擎** | 基于决策树的故障排查逻辑 | 3天 |
| **方案展示** | 给出诊断结果和解决步骤 | 1天 |
| **工单提交** | 用户填写问题描述和联系方式 | 1天 |
| **状态查询** | 查看工单处理进度 | 1天 |

#### P1 - 应该有（第一迭代）

| 功能 | 描述 | 预期工时 |
|------|------|----------|
| **知识库搜索** | 关键词检索解决方案 | 2天 |
| **智能推荐** | 根据诊断记录推荐相关文章 | 2天 |
| **用户反馈** | 对诊断结果满意度评价 | 1天 |
| **微信登录** | 基础用户体系 | 2天 |
| **通知推送** | 诊断进度/结果推送 | 1天 |

#### P2 - 可以有（第二迭代）

| 功能 | 描述 | 预期工时 |
|------|------|----------|
| **会员体系** | 会员等级、权益、解锁 | 3天 |
| **在线支付** | 急诊付费、会员购买 | 2天 |
| **预约日历** | 选择问诊时间 | 2天 |
| **历史记录** | 诊断/问诊历史 | 1天 |
| **分享功能** | 分享诊断结果 | 1天 |

#### P3 - 后续考虑

| 功能 | 描述 |
|------|------|
| **视频问诊** | 实时视频连线医生 |
| **上门服务** | 预约技术人员上门 |
| **API 接入** | 为企业客户提供 API |
| **Agent 监控** | 接入用户 Agent 实时监控 |
| **社区功能** | 用户交流互助 |

### 3.3 交互设计原则

| 原则 | 说明 | 示例 |
|------|------|------|
| **渐进式披露** | 先展示关键选项，按需展开详情 | 诊断问题先问最可能的3个原因 |
| **即时反馈** | 每个操作都有明确反馈 | 选择症状后立即高亮选中态 |
| **容错设计** | 允许用户撤销/返回上一步 | "上一步"按钮始终可见 |
| **情绪安抚** | 诊断过程中保持积极语气 | 加载时显示"虾医生正在诊断中..." |
| **进度可见** | 让用户知道在哪一步 | 进度条显示 1/5 |
