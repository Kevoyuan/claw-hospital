## 1. Claude Code 常见问题与解决方案

### 1.1 API 错误与连接问题

| 错误类型 | 错误代码 | 解决方案 |
|---------|---------|---------|
| API 速率限制 | 429 | 使用指数退避重试，批量请求，或升级到 Pro 计划 |
| 上下文窗口超限 | 400 | 将代码库拆分为更小的逻辑块，使用 `.claudeignore` 排除无关文件 |
| 认证错误 | 401/403 | 重新生成 API 密钥，验证 `CLAUDE_API_KEY` 环境变量 |
| 服务过载 | 529 | 切换到 Sonnet 模型或等待后重试 |
| 请求格式错误 | 400 | 检查 JSON 格式和必需字段 |
| 服务器错误 | 500 | 实施指数退避并重试 |

**调试技巧：**
- 使用 `/compact` 命令可减少 50% 处理时间
- 长时间对话后使用 `/clear` 开始新会话
- 为新话题开启新会话避免上下文污染

### 1.2 安装与设置问题

| 问题 | 解决方案 |
|-----|---------|
| `command not found: claude` | 检查 PATH 配置，添加 `~/.local/bin` 到 PATH |
| 权限错误 | 修复 npm 权限或使用 nvm |
| VS Code 扩展不工作 | 手动安装扩展，重启 VS Code，先运行 `/ide` 命令 |
| 启动卡顿 | 重新安装，检查工具冲突，确保 16GB+ RAM |

**PATH 配置示例：**

```bash
# macOS/Linux (Zsh)
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.zshrc
source ~/.zshrc
```

### 1.3 性能与输出质量问题

| 问题 | 原因与解决方案 |
|-----|---------------|
| 响应慢 | 保持对话聚焦，避免上传大文件，使用子代理处理复杂分析 |
| 生成"糟糕"代码 | 提供清晰详细的提示，将复杂任务拆分为小步骤 |
| 输出重复/混乱 | 使用 `/clear` 重置上下文，为新话题开新会话 |
| 幻觉 | 在提示中具体说明，使用单元测试和 lint 验证 |

## 2. AI Agent 调试技术

### 2.1 常见错误分类

#### 数据相关问题
- **数据质量差/不完整**：导致结果偏差和不可预测的模型性能
- **数据泄露**：测试数据混入训练集，导致过度乐观
- **偏见**：训练数据中的偏见导致歧视性输出
- **数据多样性不足**：模型无法有效学习

#### 模型与算法问题
- **过拟合**：在训练数据上表现好但在新数据上差
- **欠拟合**：未能捕获数据中的复杂模式
- **超参数配置错误**：导致模型性能不佳
- **幻觉**：AI 生成自信但错误的信息
- **边缘情况失败**：只能处理"理想路径"场景
- **规划器无限循环**：成功标准模糊导致重复

#### 系统与集成问题
- **工具调用错误**：AI 代理错误调用外部工具
- **内存与上下文管理**：长对话中信息丢失或内存污染
- **级联错误传播**：多步骤过程中初始错误放大
- **安全漏洞引入**：生成功能正确但有安全问题的代码

### 2.2 调试技术与解决方案

#### 系统化调试流程
1. **错误识别**：通过性能指标、用户反馈或监控工具检测异常
2. **根因分析**：确定错误来源于数据、算法还是部署
3. **修复与验证**：应用修正并用测试数据集验证
4. **监控**：部署后持续跟踪模型性能

#### AI 辅助调试
- **推理优先提示**：先让 AI 分析可能原因，再编写代码
  ```
  "列出5-7个可能的原因并为每个原因提出诊断方案，暂时不写代码"
  ```
- **自动化修复循环**：让 AI 迭代修复和测试
- **添加日志和调试语句**：让 AI 注入日志以打印输入 payload、验证输出等

#### 可观测性与监控
- **全面日志记录**：捕获每个代理操作、通信和工具使用
- **分布式追踪**：系统化记录多代理工作流中的交互
- **实时监控**：实施持续检查确保模型稳定运行

#### 增量调试技巧
- 将复杂错误拆分为更小的问题
- 使用单元测试和逐步验证
- 实现健壮的错误处理机制
- 在代码中而非提示中强制执行行为

## 3. 代码质量与架构优化

### 3.1 减少上下文需求

**核心原则**：将代码组织为问题域，而非技术层。

| 按技术层组织 | 按问题域组织 |
|------------|-------------|
| `/controllers`, `/services`, `/models` | `/auth`, `/payments`, `/notifications` |

**好处**：
- AI 只需关注当前任务相关的文件
- 减少上下文污染
- 提高代码可维护性

### 3.2 AI 作为架构审查工具

> 如果 AI 难以理解你的代码，你的同事可能也难以理解。

**实践技巧**：
- 将 API 设计为意图明确：创建专用端点而非通用端点
- 使问题域边界明确：包结构、命名约定、API 边界
- 使用清晰、直观的命名（AI 也能理解）

### 3.3 提示工程最佳实践

| 问题 | 解决方案 |
|-----|---------|
| 指令模糊 | 具体明确，提供细节 |
| 上下文管理 | 使用 `.claudeignore`，预处理大文件 |
| 输出结构混乱 | 使用 XML 标签分隔指令和上下文 |
| 复杂任务 | 拆分为步骤，使用提示链 |
| 推理不佳 | 添加"逐步思考"引导 |

## 4. 安全漏洞与防护

### 4.1 已知安全风险

- **远程代码执行 (RCE)**：恶意仓库的特殊配置文件可执行任意 shell 命令
- **API 密钥盗窃**：通过配置错误或恶意代码窃取凭证

### 4.2 安全防护措施

1. **谨慎克隆仓库**：尤其来源不信任时
2. **保护 API 密钥**：存储在环境变量或安全存储服务，绝不明文存储在代码中
3. **用户确认提示**：确保工具和服务在明确批准前不初始化
4. **企业证书配置**：如需代理 TLS 检查，设置 `NODE_EXTRA_CA_CERTS`

## GitHub Issues

### 问题1: OpenClaw 配置导致 Gateway 崩溃
- 描述: 用户在修改 `openclaw.json` 配置文件时容易导致 Gateway 崩溃，初始化向导没有询问所有必要问题导致配置不完整
- 原因: 配置文件格式错误、缺少必要字段或权限配置不正确
- 解决方案: 使用 `openclaw doctor --fix` 自动修复，运行 `openclaw status --all` 获取完整诊断报告，修改配置前备份原文件

### 问题2: OpenClaw 认证失败 (401) 和速率限制 (429)
- 描述: API 认证失败和请求速率限制是常见问题，尤其使用 Anthropic 长上下文请求时
- 原因: API 密钥无效、凭证过期、长上下文请求需要额外使用配额
- 解决方案: 使用 `openclaw models auth setup-token` 解决 401 错误；429 错误等待 60 秒后重试，添加备用模型，启用 Anthropic Extra Usage

### 问题3: OpenClaw 构建问题
- 描述: 从源码运行时不使用 pnpm 而使用 npm/yarn 会导致依赖解析问题
- 原因: 项目依赖管理使用 pnpm，其他包管理器存在兼容性问题
- 解决方案: 确保在 main 分支，使用 `pnpm install` 和 `pnpm build`

### 问题4: OpenClaw Telegram 多账户问题
- 描述: 多账户设置和消息投递存在问题
- 原因: 多账户配置复杂，API 权限设置不当
- 解决方案: 检查账户配置和 API 权限设置

### 问题5: Claude Code VS Code 扩展 v2.1.55 Windows 激活失败
- 描述: Anthropic 构建管道意外将 Linux CI 路径嵌入 Windows 扩展，导致 Windows 上激活失败
- 原因: 构建配置错误，路径包含 Linux 特定字符
- 解决方案: 升级到更新版本或降级到稳定版本

### 问题6: Claude Code 启动挂起 (v2.1.63)
- 描述: 更新后 Claude Code 在启动时挂起，原因是设置迁移问题
- 原因: 布尔标志触发无限循环，路径规范化逻辑中双斜杠前缀 (`//C:/`) 处理错误
- 解决方案: 清除设置缓存或重新安装到新位置

### 问题7: Claude Code GitHub 集成显示 0% 容量
- 描述: GitHub 集成在 Claude.ai Web 界面显示所有文件和文件夹容量为 0%，无法加载内容
- 原因: GitHub App 权限配置正确但连接存在问题
- 解决方案: 重新连接 GitHub 仓库，检查 GitHub App 权限设置

### 问题8: Claude Code 基础设施 bug 导致响应质量下降
- 描述: 2025年8月至9月期间，多个基础设施 bug 导致约 30% 用户的响应质量下降
- 原因: 请求被错误路由到错误的服务器类型，运行时性能优化导致输出损坏
- 解决方案: 等待 Anthropic 官方修复，关注状态页面更新

### 问题9: OpenClaw Gateway 服务未运行
- 描述: 服务已安装但进程无法保持运行
- 原因: 端口冲突、配置不匹配、缺少认证
- 解决方案: 检查 `openclaw gateway status`，确认 `gateway.mode="local"` 配置正确，检查端口占用

### 问题10: OpenClaw 浏览器工具失败
- 描述: 浏览器工具操作失败，即使 Gateway 本身正常
- 原因: 浏览器可执行路径无效、CDP 配置文件不可达、扩展 relay 未连接
- 解决方案: 运行 `openclaw browser start --browser-profile openclaw`，检查扩展 relay 连接状态

## 6. 社区问题收集

### 来源: Reddit (r/ChatGPT, r/LocalLLaMA, r/Artificial, r/AI_Agents)
- **问题**: AI 生成内容质量下降 - 文本过于通用、缺乏深度和真实性，内容同质化导致"香草虚空"(Vanilla Void)现象
- **解决方案**: 
  - 建立 AI 内容使用规范和社区规则
  - 开发者应优先考虑一致的功能性、透明定价和响应式人工客服
  - 为简单、重复性任务设计 AI Agent，而非"全能型"助手

- **问题**: 社交动态和真实性受到威胁 - AI 机器人冒充人类，甚至扮演敏感角色，引发信任危机
- **解决方案**: 
  - 考虑引入身份验证措施，与第三方服务合作确认用户为人类
  - 开发更先进的 AI 内容检测工具供版主使用

- **问题**: 治理和审核困难 - 志愿者版主难以区分 AI 生成内容和真人内容
- **解决方案**: 
  - 扩大 AI 特定规则的采用和执行
  - 开发更有效的 AI 内容识别和管理工具

- **问题**: 技术不稳定 - AI 工具崩溃、服务器宕机、加载失败、功能有限
- **解决方案**: 
  - 明确传达数据政策和 AI 局限性
  - 基础功能不应锁定在付费墙后

- **问题**: 安全漏洞 - AI Agent 可能导致 CRM 数据库泄露、数据外泄、跨平台级联攻击
- **解决方案**: 
  - 提高用户和开发者的 AI 素养
  - 框架应默认强制执行安全模式而非依赖开发者

- **问题**: 长项目中的上下文丢失 - AI 在复杂项目中"崩溃"，逐渐失去之前交互的记忆
- **解决方案**: 
  - 实施有效的上下文管理策略，如摘要、记忆存储和检索机制
  - 定期清理过期或臃肿的记忆文件

- **问题**: 创意任务表现差 - 难以保持"品牌调性"或理解细微差别
- **解决方案**: 
  - 使用结构化提示，包含明确问题、上下文、具体指令和示例
  - 利用 LangChain 和 LangGraph 等框架

### 来源: V2EX
- **问题**: 复杂性与配置困难 - 非技术用户发现设置过程繁琐，配置错误导致任务失败
- **解决方案**: 
  - 提供清晰详细的配置文档和逐步教程
  - 简化用户界面和配置流程
  - 提供预设模板或自动化配置脚本

- **问题**: 浏览器自动化不稳定 - 远程服务器上使用浏览器控制服务时经常超时
- **解决方案**: 
  - 优化浏览器自动化模块的稳定性和容错性
  - 提供可配置的超时设置

- **问题**: Token 成本高昂 - AI 持续"思考"和"心跳"可能迅速消耗 API 余额
- **解决方案**: 
  - 实施严格的 Token 使用监控和成本管理机制
  - 优化 Agent 决策逻辑，减少不必要的思考
  - 提供 Token 消耗预警和限制功能

- **问题**: 数据兼容性与质量问题 - 企业数据分散在不同系统，格式不兼容影响 Agent 理解上下文
- **解决方案**:

  - 利用数据集成平台集中、清洗、转换数据

- **问题**: 提示注入漏洞 - 恶意指令可能被存储、调用和执行（记忆中毒）
- **解决方案**: 
  - 实施 AI 特定安全策略，重点关注访问控制、异常检测和加密

- **问题**: AI 生成代码管理困难 - AI 倾向于将样式表统一放在一个文件中，导致难以管理
- **解决方案**: 
  - 明确指示 AI 遵循特定代码组织和样式管理规范
  - 对关键或复杂代码部分进行人工审查和重构

- **解决方案**: 
  - 优先采用可解释 AI (XAI) 技术
  - 提供决策过程的清晰文档和可视化
  - 实施全面的可观测性以便及早发现问题

- **问题**: 幻觉问题 - Agent 可能生成不正确或推测性的回复
- **解决方案**: 
  - 通过系统提示明确指示 Agent 在信息不可用时承认不确定性
  - 部署幻觉检测评估器

- **问题**: 延迟问题 - 基础设施限制和低效的 Agent 轨迹导致响应延迟
- **解决方案**: 
  - 采用缓存策略减少延迟
  - 优化 Agent 执行路径和工具编排

- **问题**: 工具选择和编排不佳 - Agent 可能无法有效选择或使用工具
- **解决方案**: 
  - 提供全面的工具描述，明确功能、输入输出要求和使用示例
  - 采用分层工具组织降低复杂性
  - 建立错误处理机制确保优雅降级

## 7. 博客问题收集 (2025-2026)

### 来源: Dev.to - How Do I Debug Failures in My AI Agents?
- 问题:
  - AI Agent 失败难以调试，因为与传统确定性软件不同，AI Agent 在概率性、多轮环境中运行
  - 长多轮交互：错误可能出现在深度对话中，使根因分析复杂
  - 涌现行为：动态协作的 Agent 可能产生意外结果
  - 不透明的推理路径：没有适当的追踪，很难理解 Agent 为何做出特定决策
  - 工具和数据依赖：工具调用或数据检索失败可能在系统中传播
- 解决方案:
  - **Agent Tracing**: 系统记录和可视化 Agent 交互、决策和状态变化
  - **可观测性平台**: 使用 Maxim AI 等平台进行分布式追踪、实时告警和自定义仪表板
  - **综合日志记录**: 捕获每个 Agent 操作、通信和工具使用
  - **交互式可视化工具**: 使用仪表板映射 Agent 轨迹，快速识别故障点
  - **集成评估管道**: 结合自动化指标和人工审查进行持续质量保证
  - **生产监控**: 持续观察 Agent 行为，设置实时告警

### 来源: Arize AI - AI Agent Debugging: Four Lessons From Shipping Alyx to Production
- 问题:
  - **注意力分散问题**: Agent 容易在多任务处理中迷失，忘记原始用户请求，陷入工具调用循环
  - **上下文洪水**: 工具输出、JSON blob 和中间结果淹没上下文窗口，原始请求被埋没
  - **过早调用 finish**: Agent 在完成任务前过早调用结束工具
  - **规划循环**: Agent 反复 reorganizing 自己的待办事项列表，27 次 LLM 调用却从未真正完成任何事
  - **大数据集处理**: 大型实验数据可能轻松超过 100,000 tokens，导致上下文窗口溢出
- 解决方案:
  - **将规则写入代码**: 提示是建议，代码是约束。使用工具验证来强制执行规则
  - **创建规划优先工具**: 引入 todo_write、todo_update、todo_read 工具，在 Agent 处理任何数据前强制其制定计划
  - **计划状态粒度**: 使用 4 种状态（pending、in_progress、completed、blocked）而非 2 种
  - **计划存储在对话历史外**: 将计划存储在磁盘上，每次 LLM 调用时动态重新生成 PlanMessage 并注入系统提示之后
  - **Finish Gate**: 如果 Agent 尝试用未完成任务调用 finish，抛出 RecoverableException 错误
  - **大 JSON 处理**: 使用 LargeJson 模式，将数据存储在带外，给 Agent 一个句柄而不是全部塞入上下文
  - **Few-shot 示例有效**: 展示具体示例比抽象指令更有效

### 来源: FutureAGI - Comprehensive Guide to Debugging AI Agents in 2025
- 问题:
  - **复杂性和非确定性**: 与确定性软件不同，AI Agent 以概率方式运行，传统调试技术不足
  - **涌现行为**: 多 Agent 系统中的动态协作可能导致难以追踪的结果
  - **工具调用失误**: Agent 可能向 API 发送模糊指令、依赖过时模式或遗漏必需参数
  - **幻觉级联**: 自信但错误的陈述可能滚雪球成重大问题
  - **内存膨胀与状态漂移**: 随着时间推移，向量存储可能膨胀导致响应不一致和延迟增加
  - **非确定性输出漂移**: 同一请求可能在生产中产生不同答案
  - **Agent 间对齐错误**: 多 Agent 系统中，一个 Agent 的模糊或错误指令可能导致另一个出错
  - **AI 生成错误**: AI 调试工具本身可能产生不正确建议或引入安全漏洞
- 解决方案:
  - **全面可观测性实现**: 从第一天起实施结构化日志、分布式追踪
  - **交互式可视化工具**: 利用 Agent Compass、Maxim AI、Arize AI、LangSmith、WandB 等平台
  - **AI 驱动的调试工具**: 使用自动错误检测、诊断和修复建议的工具
  - **构建主动调试循环**: 在 Agent 在生产中崩溃前建立调试工具、技能和可观测性机制
  - **代码约束而非提示约束**: 工具验证提供具体约束
  - **提供足够上下文**: 对上下文进行精确处理，只包含当前任务相关内容
  - **将良好行为结晶为测试**: 当 Agent 表现出期望行为时，将其捕获并 formalize 为测试
  - **沙盒化一切**: 容器化调试运行以防止潜在损害
  - **保持人在循环中**: 要求对 AI 生成的修复进行人工批准
