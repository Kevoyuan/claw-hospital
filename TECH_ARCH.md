## 1. 概述

### 1.1 项目背景

OpenClaw 维修医院是一个面向 AI Agent 系统的智能化故障诊断与维修服务平台。该平台旨在为 OpenClaw 用户提供自动化的故障检测、诊断建议、维修指导以及预防性维护能力。

### 1.2 设计目标

| 目标 | 描述 |
|------|------|
| **高可用性** | 系统可用性达到 99.9% 以上，支持多区域部署 |
| **智能化诊断** | 基于 AI 的故障根因分析，提供精准诊断建议 |
| **实时响应** | 毫秒级故障检测与告警响应 |
| **可扩展性** | 支持水平扩展，满足业务增长需求 |
| **安全性** | 企业级安全防护，保障数据隐私 |

### 2.2 用户服务 (User Service)

#### 2.2.1 功能职责

用户服务是整个系统的基础服务，负责用户身份认证、授权管理、个人信息维护等核心功能。

#### 2.2.2 核心功能

| 功能模块 | 描述 | API 端点 |
|---------|------|----------|
| **认证模块** | 用户注册、登录、登出、Token 签发、刷新 | /api/v1/auth/* |
| **授权模块** | RBAC 角色权限管理、访问控制 | /api/v1/permissions/* |
| **用户管理** | 用户 CRUD、批量操作、组织架构 | /api/v1/users/* |
| **会话管理** | 在线状态、会话追踪、设备管理 | /api/v1/sessions/* |
| **个人设置** | 主题、语言、通知偏好 | /api/v1/settings/* |

#### 2.2.3 技术实现

用户服务核心接口定义：

- register(dto: RegisterDTO): Promise<User> - 用户注册
- login(dto: LoginDTO): Promise<AuthToken> - 用户登录
- logout(token: string): Promise<void> - 用户登出
- refreshToken(refreshToken: string): Promise<AuthToken> - 刷新令牌
- getUserById(id: string): Promise<User> - 获取用户信息
- updateUser(id: string, dto: UpdateUserDTO): Promise<User> - 更新用户
- deleteUser(id: string): Promise<void> - 删除用户
- assignRole(userId: string, roleId: string): Promise<void> - 分配角色
- revokeRole(userId: string, roleId: string): Promise<void> - 撤销角色
- getUserPermissions(userId: string): Promise<Permission[]> - 获取权限

### 2.3 诊断服务 (Diagnosis Service)

#### 2.3.1 功能职责

诊断服务是系统的核心智能服务，负责收集系统指标、分析故障模式、提供诊断建议。

#### 2.3.2 核心功能

| 功能模块 | 描述 | 优先级 |
|---------|------|--------|
| **指标采集** | 从各节点收集系统指标、性能数据 | P0 |
| **异常检测** | 基于规则和 ML 的异常识别 | P0 |
| **根因分析** | RCA (Root Cause Analysis) 智能分析 | P0 |
| **健康评分** | 系统整体健康度评估 | P1 |
| **趋势预测** | 基于时序数据的故障预测 | P1 |
| **诊断报告** | 自动生成诊断报告 | P2 |

#### 2.3.3 诊断流程

数据采集 → 异常检测 → 根因分析 → 告警生成/建议生成 → 诊断报告输出

### 2.4 工单服务 (Ticket Service)

#### 2.4.1 功能职责

工单服务管理从故障发现到修复完成的全流程工单，支持多人协作、自动分配、状态追踪。

#### 2.4.2 工单状态机

新建 → 已确认 → 处理中 → 已解决 → 已关闭

- 新建工单后进入已确认状态
- 处理中状态可重新打开
- 已解决后可关闭

#### 2.4.3 SLA 管理

| 优先级 | 响应时间 | 解决时间 | 升级阈值 |
|--------|----------|----------|----------|
| P0 (紧急) | 5 分钟 | 1 小时 | 15 分钟 |
| P1 (高) | 15 分钟 | 4 小时 | 1 小时 |
| P2 (中) | 1 小时 | 24 小时 | 4 小时 |
| P3 (低) | 4 小时 | 72 小时 | 24 小时 |

### 2.5 知识库服务 (Knowledge Service)

#### 2.5.1 功能职责

知识库服务存储和管理故障案例、解决方案、操作手册等知识资产，为 AI 诊断提供知识支持。

#### 2.5.2 知识库结构

- 故障案例库：症状描述、根因分析、解决方案、相关指标
- 解决方案库：问题类型、操作步骤、预期结果、注意事项
- 操作手册：系统配置、运维指南、应急处理
- FAQ：常见问题、标准答案

## 3. AI 诊断引擎技术方案

### 3.1 架构设计

AI 诊断引擎采用多层架构设计：

1. **数据输入层**：指标数据、日志数据、追踪数据、事件数据
2. **特征工程层**：数据清洗、特征提取、特征选择、特征变换、特征存储
3. **模型推理层**：
   - 异常检测模型 (Isolation Forest)
   - 根因分析模型 (LLM + 知识图谱)
   - 预测模型 (LSTM/Transformer)
4. **决策输出层**：诊断结论、修复建议、置信度评分

### 3.2 核心模型设计

#### 3.2.1 异常检测模型

**技术选型：** Isolation Forest + LSTM 混合架构

采用多层异常检测架构：
- 第一层：统计异常检测 (Z-score, IQR)
- 第二层：Isolation Forest 无监督异常检测
- 第三层：LSTM 时序异常检测

通过投票机制融合各层检测结果。

#### 3.2.2 根因分析模型 (RCA)

**技术选型：** 大语言模型 (LLM) + 知识图谱混合架构

基于知识图谱的根因分析流程：
1. 提取异常事件
2. 构建故障传播图
3. 使用图算法识别根因节点
4. LLM 生成根因解释

#### 3.2.3 预测模型

**技术选型：** Transformer-based 时序预测

基于 Transformer 的时序故障预测：
- 多头注意力机制捕捉长距离依赖
- 预测未来时间窗口内的故障概率

### 3.3 AI 流水线设计

```
数据输入 → 模型推理 → 结果输出 → 动作执行
   ↓           ↓            ↓          ↓
指标采集   异常检测    诊断分析    生成告警
   ↓           ↓            ↓          ↓
实时流处理  根因分析    建议生成    创建工单

Pipeline 延迟目标: < 500ms
```

### 3.4 模型服务化

使用 Kubernetes 上的 KServe 进行模型服务化部署，支持：
- 自动扩缩容
- GPU 加速推理
- 模型版本管理
- A/B 测试

## 4. 用户系统设计

### 4.1 认证系统

#### 4.1.1 认证流程

1. 用户发起登录请求
2. 返回认证页面/Redirect
3. 用户填写凭证
4. 提交凭证验证
5. 验证凭证，返回 Token
6. 携带 Token 访问资源
7. 验证 Token，返回资源

#### 4.1.2 Token 设计

| Token 类型 | 有效期 | 用途 | 存储 |
|------------|--------|------|------|
| Access Token | 15 分钟 | API 访问 | Memory (客户端) |
| Refresh Token | 7 天 | 续期 Access Token | HttpOnly Cookie |
| ID Token | 15 分钟 | 用户身份信息 | Memory (客户端) |

JWT Access Token 结构包含：
- Header: alg (RS256), typ (JWT)
- Payload: sub (用户ID), aud (受众), iss (签发者), iat (签发时间), exp (过期时间), jti (Token唯一标识)
- Custom Claims: roles (用户角色), permissions (权限列表), tenant_id (租户ID)

### 4.2 权限系统 (RBAC + ABAC)

#### 4.2.1 角色模型

采用 RBAC 模型：
- 权限 (Permission) - 基础权限单元
- 角色权限 (RolePerm) - 角色与权限的多对多关系
- 角色 (Role) - 角色集合
- 用户 (User) - 拥有角色的主体

#### 4.2.2 预定义角色

| 角色 | 描述 | 权限 |
|------|------|------|
| **超级管理员** | 系统全权管理 | 所有权限 |
| **租户管理员** | 租户内管理 | 用户管理、工单管理、查看诊断 |
| **运维工程师** | 日常运维 | 查看诊断、处理工单、配置告警 |
| **开发人员** | 开发调试 | 查看诊断、查看日志 |
| **只读用户** | 查看报表 | 只读访问 |

#### 4.2.3 权限检查实现

使用装饰器进行权限检查：
```typescript
@RequirePermission('diagnosis:read')
async function getDiagnosis(ctx: Context, diagnosisId: string) {
  const hasPermission = await permissionService.check(
    ctx.user.id,
    'diagnosis:read',
    { resource: diagnosisId, resourceType: 'diagnosis' }
  );
  
  if (!hasPermission) {
    throw new ForbiddenError('无诊断报告读取权限');
  }
  
  return await diagnosisService.getById(diagnosisId);
}
```

### 4.3 多租户设计

多租户架构支持三种模式：

- **租户 A (独栋医院)**：专属数据库、专属 AI 模型、定制 UI
- **租户 B (连锁诊所)**：共享数据库、租户隔离、标准功能
- **租户 C (个人工作室)**：共享实例、基础功能、按量计费

共享层包括：
- AI 诊断引擎 (共享模型训练，隔离推理)
- 知识库 (基础库共享，专业库隔离)
- 计费系统

## 5. 数据存储方案

### 5.1 存储架构

采用多数据源架构：

| 存储类型 | 用途 | 特点 |
|----------|------|------|
| PostgreSQL | 业务数据、用户、工单、诊断记录 | 事务支持、复杂查询 |
| Redis | 缓存/会话、热点数据、限流计数 | 高速读写、低延迟 |
| Elasticsearch | 日志/搜索、诊断日志、全文检索 | 全文搜索、聚合分析 |
| InfluxDB | 时序指标、监控数据、性能数据 | 时序压缩、高写入 |
| Kafka | 消息队列、事件流、数据同步 | 消息持久化、流处理 |
| S3/MinIO | 文件/图片、模型文件 | 海量存储、成本低 |

### 5.2 数据库设计

#### 5.2.1 PostgreSQL 核心表结构

**用户表 (users)**
- id: UUID 主键
- email: 唯一邮箱
- username: 唯一用户名
- password_hash: 密码哈希
- tenant_id: 租户外键
- role_id: 角色外键
- status: 状态
- created_at/updated_at: 时间戳
- last_login_at: 最后登录时间
- metadata: JSONB 元数据

**租户表 (tenants)**
- id: UUID 主键
- name: 租户名称
- slug: 唯一标识
- plan: 套餐类型
- status: 状态
- settings: JSONB 配置
- expires_at: 到期时间

**诊断记录表 (diagnosis_records)**
- id: UUID 主键
- tenant_id/user_id: 关联外键
- target_id/target_type: 诊断对象
- health_score: 健康分 (0-100)
- anomaly_level: 异常级别 (normal/warning/critical)
- root_cause: 根因描述
- recommendations: JSONB 建议
- metrics_snapshot: JSONB 指标快照
- logs_excerpt: 日志摘要
- created_at/resolved_at: 时间戳

**工单表 (tickets)**
- id: UUID 主键
- tenant_id: 租户外键
- title/description: 标题描述
- priority: 优先级 (P0-P3)
- status: 状态
- reporter_id/assignee_id: 报告人/处理人
- diagnosis_id: 关联诊断
- first_response_at/resolved_at/closed_at: 时间戳
- sla_deadline: SLA 截止时间

**索引优化**
- idx_diagnosis_tenant_time: 租户+时间复合索引
- idx_diagnosis_target: 诊断对象索引
- idx_ticket_status: 状态优先级索引
- idx_ticket_assignee: 处理人状态索引

#### 5.2.2 时序数据 (InfluxDB)

**system_metrics 表**
- Tags: tenant_id, node_id, metric_type
- Fields: cpu_usage, memory_usage, disk_usage, network_in/out, active_connections, error_count, latency_p50/p99

**diagnosis_results 表**
- Tags: tenant_id, target_id, target_type, anomaly_level
- Fields: health_score, confidence, anomaly_score

### 5.3 缓存策略

| 数据类型 | 缓存策略 | TTL | 淘汰策略 |
|----------|----------|-----|----------|
| 用户会话 | Redis String | 15 分钟 | LRU |
| 用户信息 | Redis Hash | 1 小时 | LRU |
| 诊断结果 | Redis String | 5 分钟 | LFU |
| 权限缓存 | Redis String | 10 分钟 | TTL |
| 热点工单 | Redis List | 30 分钟 | LRU |
| 限流计数 | Redis String | 1 分钟 | TTL |

缓存服务采用多级缓存：本地内存 + Redis，通过消息队列实现分布式缓存更新。

## 6. 扩展性设计

### 6.1 水平扩展策略

采用微服务架构，每个服务可独立水平扩展：

- **无状态服务**：通过负载均衡器分发请求，支持服务实例动态增减
- **有状态服务**：使用分布式存储 (etcd/Consul) 管理状态，支持数据分片
- **数据库扩展**：
  - 读请求：主从复制 + 读写分离
  - 写请求：分库分表 (按租户ID/用户ID)
- **缓存扩展**：Redis Cluster 集群模式，支持数据自动分片

### 6.2 服务治理

**核心能力：**

1. **服务注册与发现**
   - 基于 Consul/etcd 的服务注册中心
   - 健康检查 + 自动摘除故障节点
   - DNS/VIP 方式的服务发现

2. **负载均衡**
   - 轮询、加权、最少连接
   - 地理位置感知路由
   - 金丝雀发布支持

3. **熔断与降级**
   - 熔断器模式：快速失败，防止级联故障
   - 降级策略：返回缓存/默认值
   - 熔断恢复：自动/手动

4. **限流**
   - 多维度限流：用户/租户/IP
   - 限流算法：令牌桶、滑动窗口
   - 限流策略：排队/拒绝/降级
